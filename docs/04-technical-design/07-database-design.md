# 07. Database Design

## Purpose

This chapter defines the logical and physical database architecture for the KrewOps Recruitment Platform, including entity ownership, schema design, relationships, indexing strategy, transactions, migrations, and performance considerations.

---

# Database Technology

- Primary Database: PostgreSQL
- Cache: Redis
- Object Storage: S3-compatible storage for resumes and attachments

---

# Database Architecture

The platform follows a service-owned data model. Each microservice owns its tables and accesses only its own schema. Cross-service data sharing is achieved through REST APIs and domain events rather than direct database access.

---

# Core Business Entities

| Entity | Owning Service |
|---|---|
| User | Authentication Service |
| Organization | Organization Service |
| Job | Job Service |
| Candidate | Candidate Service |
| Application | Candidate Service |
| Interview | Interview Service |
| Notification | Notification Service |
| Audit Log | Shared Infrastructure |

---

# Entity Relationships

- One Organization can own many Jobs.
- One Job can receive many Applications.
- One Candidate can apply to many Jobs.
- One Application can have multiple Interview rounds.
- One User belongs to one or more Organizations.

---

# Primary Keys

- UUID used for all business entities.
- Immutable identifiers.
- Generated by the application layer.

---

# Constraints

- Primary Key constraints
- Foreign Key constraints where applicable
- Unique constraints for business identifiers
- NOT NULL constraints for mandatory attributes
- Check constraints for domain validation

---

# Indexing Strategy

Indexes shall be created for:

- Primary keys
- Foreign keys
- Frequently filtered columns
- Frequently sorted columns
- Searchable business identifiers

Composite indexes should be used for common query patterns.

---

# Transaction Management

- ACID transactions
- Optimistic locking where applicable
- Short-lived transactions
- Rollback on failure

---

# Database Migration

Schema changes shall be managed using version-controlled migration scripts executed during deployment.

---

# Backup and Recovery

- Automated scheduled backups
- Point-in-time recovery support
- Disaster recovery validation
- Periodic restore testing

---

# Performance Considerations

- Query optimization
- Proper indexing
- Connection pooling
- Pagination for large datasets
- Archive historical data when required

---

# Summary

The database design provides a scalable, secure, and maintainable persistence architecture with clear ownership, optimized access patterns, and controlled schema evolution.
